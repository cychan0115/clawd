# Agent Health Check Skill

## Purpose

Check the health of CY's multi-agent Jira workflow between Anna / Hermes and huahua / OpenClaw.
Jira is the source of truth.
This skill generates a human-readable report for CY and ChatGPT review.

> **身份限制**：
> - **Anna (本地 MBP)**：执行全局 Health Check，检查所有 ticket 和 Agent 状态
> - **huahua (mini2)**：只能报告自身状态，不能做全局验收或 Review

## Inputs

- Jira project key, usually `AIWH`
- Optional ticket range
- Optional time window
- Optional target agent, e.g. `huahua`
- Optional executor ID, e.g. `huahua-mini2`
- Optional status filter, e.g. Backlog / 处理中 / Done / Blocked

## Checks

Anna must check:
1. Tickets in Backlog but not claimed for too long.
2. Tickets in `处理中` without `[LEASE]`.
3. Tickets with `[LEASE]` but no `[Preflight Check]`.
4. Tickets where Preflight is `CANNOT START` but execution continued.
5. Tickets with expired Lease.
6. Tickets blocked by unclear requirements.
7. Tickets blocked by dependencies not `[ACCEPTED]`.
8. Tickets Done but not reviewed by Anna.
9. Tickets with Done Report but no evidence.
10. Tickets with `[ACCEPTED]` missing after Done Report.
11. Downstream Tickets started before upstream Tickets were `[ACCEPTED]`.
12. Duplicate claims by multiple executors.
13. Tickets missing Target Agent / Owner Agent / Reviewer Agent.
14. Tickets that should become Skill / SPEC / script / automation.
15. Tickets whose Jira assignee conflicts with Owner Agent / Executor ID.
16. Tickets where local `AGENT_STATUS.md` conflicts with Jira state.

## Rules

- Do not guess missing state.
- If Jira data is incomplete, mark it as UNKNOWN.
- If a Ticket has conflicting signals, mark health as CONFLICT.
- If a Ticket is unsafe to continue, mark health as CRITICAL.
- If a Ticket is only waiting for CY or Anna, mark health as BLOCKED, not FAILED.
- If a Ticket is waiting for review after Done Report, Anna must not create downstream execution Tickets until it is `[ACCEPTED]`.
- Reports are generated from Jira state and structured comments; reports are not the source of truth.
- Never include secrets, API tokens, bearer tokens, cookies, private keys, or raw production credentials in reports.

## Minimal Procedure

1. Read live Jira state for the project or requested ticket range.
2. Export raw JSON snapshot when useful for debugging.
3. Parse status, assignee, description fields, dependencies, and recent comments.
4. Identify Lease, Preflight, Progress Update, Blocker, Done Report, Review, Accepted, and Rework comments.
5. Compare actual state to the checks above.
6. Generate `reports/AGENT_HEALTH_REPORT_<date>.md`.
7. Summarize the top risks and suggested actions for CY.
